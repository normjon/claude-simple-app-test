# User Profile API

A TypeScript/Node.js REST API for user management, deployed on AWS EKS with Karpenter auto-scaling, DynamoDB persistence, and ALB path-based routing.

---

## Architecture

```
Internet (HTTP:80)
       │
       ▼
  ALB — shared, internet-facing (group: platform)
       │  Listener rule: path /user-app*
       ▼
  Kubernetes Ingress (ingressClassName: alb)
       │
       ▼
  ClusterIP Service  :80
       │
       ▼
  Pod (Express :3000)          ←── IRSA credentials
  /user-app/health                   │
  /user-app/users              IAM Role: ex-karpenter-user-app
  /user-app/users/:id                │
       │                             ▼
       └──────────────────►  DynamoDB: app-users
```

**Key design decisions:**

| Decision | Reason |
|---|---|
| **APP_BASE_PATH env var** | AWS ALB does not strip path prefixes. The app mounts all routes under `process.env.APP_BASE_PATH` so `/user-app/*` routes correctly without any ALB rewrite rules. |
| **Shared ALB (IngressGroup)** | All apps in the cluster share one ALB via `group.name: platform`, reducing cost. Each app owns its own path prefix. |
| **arm64 (Graviton) image** | Karpenter provisions arm64 spot instances. Building amd64 produces an `exec format error` at runtime. |
| **Strict pod anti-affinity** | One pod per AZ forces Karpenter to provision EC2 nodes across all 3 AZs for true HA. |
| **maxSurge: 0 / maxUnavailable: 1** | Strict anti-affinity leaves no room for a surge pod during rollouts. Terminating one old pod first (freeing its AZ slot) prevents scheduling deadlocks. |
| **IRSA** | IAM Roles for Service Accounts — no credentials in code or environment. The AWS SDK resolves them automatically from the pod's projected service account token. |
| **Git SHA image tags** | Unique per-commit tags prevent stale image cache across Karpenter nodes and make rollbacks unambiguous. |
| **Explicit ALB health check path** | AWS ALB defaults its target group health check to `/`, which returns 404 on this app. The Ingress declares `alb.ingress.kubernetes.io/healthcheck-path: /user-app/health` explicitly. All three health signals (ALB target group, K8s liveness probe, K8s readiness probe) are driven from `values.ingress.basePath` as the single source of truth. |

- Node.js 20+
- Docker (with `docker buildx` for cross-platform builds)

- AWS CLI + credentials (`awssandbox` for sandbox environments)

### Quick Start

```bash
npm install
npm run dev        # hot-reload dev server on :3000
```

> **Path prefix in local dev:** `APP_BASE_PATH` is not set when running `npm run dev`, so routes are at `/health`, `/users`, `/users/:id` — no prefix. The prefix is only active in EKS (injected by the Helm chart). Tests always use the prefixed paths via `src/test-setup.ts`.

### Local Dev with DynamoDB Local

`docker-compose.yml` starts DynamoDB Local and automatically creates the `app-users` table. Data persists in a named volume across restarts.

```bash
# Start DynamoDB Local (table is created automatically)
docker compose up -d

# Start dev server pointed at local DynamoDB
DYNAMODB_ENDPOINT=http://localhost:8000 npm run dev
```

```bash
# Health (no DynamoDB needed)
curl http://localhost:3000/health

# Create a user
curl -X POST http://localhost:3000/users \
  -H "Content-Type: application/json" \
  -d '{"email": "test@example.com", "name": "Test User"}'

# Search
curl "http://localhost:3000/users?email=test@example.com"
```

```bash
# Stop (preserves data volume)
docker compose down

# Full reset (deletes data)
docker compose down -v
```

### Viewing Local Logs

```bash
# Dev server — logs appear in the same terminal (structured JSON)
npm run dev

# Running Docker container
docker logs -f <container-id>
```

### Common Local Errors

| Error | Cause | Fix |
|---|---|---|
| `ECONNREFUSED` on DynamoDB calls | DynamoDB Local not running | `docker compose up -d` and set `DYNAMODB_ENDPOINT=http://localhost:8000` |
| `Cannot do operations on a non-existent table` | Table lost after restart | `docker compose up -d` — init service recreates the table |
| `Missing credentials` | No AWS credentials | `awssandbox` or `aws configure` |
| Port 3000 in use | Another process | `PORT=3001 npm run dev` |

## API Reference

All routes are mounted under `APP_BASE_PATH`. In EKS that is `/user-app`; locally (no env var set) there is no prefix.

| Method | Path | Description |
|---|---|---|
| `GET` | `/health` | Health check — no DynamoDB required |
| `POST` | `/users` | Create a new user |
| `GET` | `/users` | Search/list users (up to 100) |
| `GET` | `/users/:id` | Get a user by ID |

Allowed filter attributes for `GET /users`: `userId`, `email`, `name`. Each accepts a single scalar value. Multiple filters are AND-ed together.

### Requests and Responses

**Health check**
```bash
curl http://localhost:3000/health
# {"status":"ok","timestamp":"2026-01-01T00:00:00.000Z"}
```

**Create user**
```bash
curl -X POST http://localhost:3000/users \
  -H "Content-Type: application/json" \
  -d '{"email": "user@example.com", "name": "John Doe"}'
# 201: {"userId":"<uuid>","email":"user@example.com","name":"John Doe"}
```

**Get by ID**
```bash
curl http://localhost:3000/users/<userId>
# 200: {"userId":"...","email":"...","name":"..."}
# 404: {"error":"User not found"}
```

**Search — single filter**
```bash
curl "http://localhost:3000/users?email=user@example.com"
curl "http://localhost:3000/users?name=John%20Doe"
curl "http://localhost:3000/users?userId=<userId>"
```

**Search — combined AND filter**
```bash
curl "http://localhost:3000/users?name=John%20Doe&email=user@example.com"
```

**Search — return all (up to 100)**
```bash
curl http://localhost:3000/users
# {"users":[...]}
```

**Errors**
```bash
curl "http://localhost:3000/users?role=admin"
# 400: {"error":"Unsupported query parameter: role"}

curl "http://localhost:3000/users?email=a@b.com&email=c@d.com"
# 400: {"error":"Query parameter \"email\" must be a single value"}
```

## Testing

### Approach

Tests follow strict TDD (Red → Green → Refactor). DynamoDB is fully mocked via `aws-sdk-client-mock` — no AWS credentials or running infrastructure needed.

`src/test-setup.ts` sets `process.env.APP_BASE_PATH = '/user-app'` before any module loads (via `jest.config.js` `setupFiles`). This means **all tests run against the prefixed routes**, matching production behaviour exactly.

### Running Tests

```bash
npm test          # watch mode (development)
npm run test:ci   # single run with coverage (CI / before building)
npm run lint      # ESLint
```

### Adding a New Endpoint

Follow TDD strictly:

1. **RED** — write a failing test in the appropriate `*.test.ts` file using the prefixed path (e.g. `/user-app/users`). Run `npm test` and confirm it fails.
2. **GREEN** — add the route to `src/app.ts` on the `router` (not directly on `app`). Run `npm test` and confirm it passes.
3. **REFACTOR** — clean up. Run `npm test` again to confirm nothing regressed.
4. Update `openapi.yaml` to reflect the new endpoint.

> **Important:** All routes must be registered on `router`, not on `app` directly. `app.use(BASE_PATH, router)` at the bottom of `app.ts` is what applies the prefix.

## CI/CD Pipeline & Change Control

All changes flow through GitHub Actions. The pipeline enforces: tests pass on every PR → merge to `main` → image built and deployed automatically.

> **Current state:** single environment (dev). The `gitops-config/` folder is structured for multi-environment expansion — `test/`, `staging/`, and `prod/` overrides are scaffolded but not yet active. When dedicated infrastructure is provisioned for those environments, add a `promote.yml` workflow and GitHub Environment approval gates.

### Pipeline Flow

```
feature/* → Pull Request → [ci.yml: tests + lint must pass] → merge to main
                                                                      │
                                                              deploy-dev.yml
                                                                      │
                                                             build arm64 image
                                                             push to ECR
                                                             helm upgrade
                                                             smoke test /health
                                                                      │
                                                                    [dev]
```

### Workflows

#### `ci.yml` — Pull Request gate
Runs on every PR targeting `main`. The PR cannot be merged if this fails.
- `npm ci` — clean install
- `npm run test:ci` — full test suite with coverage
- `npm run lint` — ESLint

#### `deploy-dev.yml` — Auto-deploy on merge
Runs automatically on every merge to `main`.
1. Authenticates to AWS via OIDC (`AWS_DEPLOY_ROLE_ARN` secret — no stored credentials)
2. Builds `linux/arm64` image using Docker Buildx
3. Tags with `git rev-parse --short HEAD` and pushes to ECR
4. Helm upgrades using `gitops-config/dev/values.yaml`
5. Smoke-tests `/user-app/health` through the ALB

### One-Time Setup (GitHub Admin)

#### 1. Configure GitHub Environment

Go to **Settings → Environments** and create a `dev` environment with no required reviewers.

#### 2. Add Repository Secret

Go to **Settings → Secrets and variables → Actions** and add:

| Secret | Value |
|---|---|
| `AWS_DEPLOY_ROLE_ARN` | ARN of the IAM role GitHub Actions assumes via OIDC (provisioned by infra-repo) |

The IAM role trust policy must allow `token.actions.githubusercontent.com` as the OIDC provider, restricted to `repo:normjon/claude-simple-app-test:*`.

#### 3. Branch Protection on `main`

Go to **Settings → Branches → Add rule** for `main`:
- Require status checks: `Test & Lint` (from `ci.yml`)
- Require pull request reviews: 1 approver minimum
- Do not allow force pushes

### Expanding to Multiple Environments

When additional infrastructure is ready, the expansion path is:
1. Provision DynamoDB table, IRSA role, and EKS namespace per environment
2. Update `gitops-config/{env}/values.yaml` with real ARNs
3. Add a `promote.yml` workflow (`workflow_dispatch` with `image_tag` + `target_environment` inputs)
4. Create GitHub Environments with approval gates: test (1 reviewer), staging (2), prod (2)
5. One release in flight at a time — do not merge to `main` until the previous SHA has cleared or been abandoned

## Manual Release (Emergency / Local Debugging Only)

> The pipeline above is the standard release path. Use the manual steps below only for emergency hotfixes or when debugging deployment issues locally. All production changes should still go through a PR and the pipeline.

This is the full sequence from a code change to a live update in EKS. Complete every step in order — do not skip.

### Step 1 — Tests

```bash
npm run test:ci
```

All 16+ tests must pass. If any fail, fix them before proceeding.

### Step 2 — ECR Authentication

```bash
aws ecr get-login-password --region us-east-2 \
  | docker login --username AWS --password-stdin \
    096305373014.dkr.ecr.us-east-2.amazonaws.com
```

> Use `awssandbox` first if credentials have expired.

### Step 3 — Build (linux/arm64)

```bash
IMAGE=096305373014.dkr.ecr.us-east-2.amazonaws.com/user-app
TAG=$(git rev-parse --short HEAD)

docker build --platform linux/arm64 -t ${IMAGE}:${TAG} .
```

Verify architecture:
```bash
docker inspect ${IMAGE}:${TAG} --format '{{.Architecture}}'
# Must be: arm64
```

> **Never omit `--platform linux/arm64`.** An amd64 image produces `exec format error` at runtime on Graviton nodes.
> **Never reuse a tag.** Always use the git SHA — unique tags prevent stale image caches on Karpenter nodes.

### Step 4 — Push to ECR

```bash
docker push ${IMAGE}:${TAG}
```

### Step 5 — Configure kubectl

```bash
aws eks --region us-east-2 update-kubeconfig --name ex-claude-cloud-test
```

Verify pre-flight:
```bash
kubectl config current-context        # must point at ex-karpenter
kubectl get nodepools                 # must return at least one NodePool
kubectl get ec2nodeclasses            # must return at least one EC2NodeClass
```

### Step 6 — Deploy via Helm

```bash
helm upgrade --install user-app ./charts/user-app \
  --set image.repository=${IMAGE} \
  --set image.tag=${TAG} \
  --wait \
  --timeout 5m
```

`--wait` blocks until all 3 replicas pass their readiness probe (`/user-app/health`). On first install, Karpenter provisions 3 EC2 nodes (one per AZ) — expect 2–3 minutes before pods are ready.

### Step 7 — Verify

```bash
# All 3 pods Running, one per AZ
kubectl get pods -l app.kubernetes.io/name=user-app -o wide

# Karpenter provisioned nodes
kubectl get nodeclaims

# Helm release is deployed
helm list

# Ingress has an ALB address
kubectl get ingress user-app

# No errors in LBC events
kubectl describe ingress user-app | grep -A 5 Events
```

### Step 8 — Smoke Test via ALB

```bash
ALB=$(kubectl get ingress user-app -o jsonpath='{.status.loadBalancer.ingress[0].hostname}')

curl http://${ALB}/user-app/health

curl -X POST http://${ALB}/user-app/users \
  -H "Content-Type: application/json" \
  -d '{"email": "smoke@example.com", "name": "Smoke Test"}'

curl "http://${ALB}/user-app/users?email=smoke@example.com"
```

## Known Issues & Troubleshooting

### DynamoDB — "Internal server error" on all operations

**Symptom:** Every call except `/health` returns `{"error":"Internal server error"}`. Pod logs:
```json
{"level":"error","message":"Failed to create user","error":"Not authorized to perform sts:AssumeRoleWithWebIdentity"}
```

**Cause:** IRSA misconfiguration. Most common root cause: the IAM role trust policy `sub` condition uses the plural `system:serviceaccounts:...` instead of the singular `system:serviceaccount:...`. AWS `StringEquals` is exact — the mismatch silently rejects every token.

**Diagnose:**
```bash
# Decode the JWT sub claim from a running pod
POD=$(kubectl get pods -l app.kubernetes.io/name=user-app -o jsonpath='{.items[0].metadata.name}')
kubectl exec $POD -- sh -c 'cat $AWS_WEB_IDENTITY_TOKEN_FILE' \
  | cut -d. -f2 | base64 -d 2>/dev/null | python3 -m json.tool | grep -E 'sub|aud'
# Expected: "sub": "system:serviceaccount:default:user-app-sa"

# Check the trust policy condition
aws iam get-role --role-name ex-karpenter-user-app \
  --query 'Role.AssumeRolePolicyDocument' --output json
```

**Fix:** Update the IAM trust policy `sub` condition to singular `serviceaccount`. This is an infra-repo change.

### exec format error

**Symptom:** Pods in `CrashLoopBackOff`. `kubectl logs` shows:
```
exec /usr/local/bin/docker-entrypoint.sh: exec format error
```

**Cause:** Image built for amd64. Karpenter nodes are arm64 (Graviton).

**Fix:**
```bash
docker build --platform linux/arm64 -t ${IMAGE}:${TAG} .
docker inspect ${IMAGE}:${TAG} --format '{{.Architecture}}'   # must be arm64
docker push ${IMAGE}:${TAG}
```

### Pods stuck in Pending

**Symptom:** All pods remain `Pending`. `kubectl describe pod` shows:
```
0/N nodes are available: N node(s) had untolerated taint {eks.amazonaws.com/compute-type: fargate}
```

**Cause:** No Karpenter NodePool or EC2NodeClass exists. Only Fargate nodes are present.

**Fix:**
```bash
kubectl get nodepools        # must return resources
kubectl get ec2nodeclasses   # must return resources
```
If missing, the infra-repo must apply the Karpenter configuration.

### ALB not routing traffic / 404 from ALB

**Symptom:** `curl http://${ALB}/user-app/health` returns nothing or 404.

**Diagnose:**
```bash
# Check Ingress has an address and no error events
kubectl describe ingress user-app

# Check LBC is running
kubectl get pods -n kube-system -l app.kubernetes.io/name=aws-load-balancer-controller

# Check target group health via AWS console or CLI
# All 3 pod IPs should be healthy targets
```

**Common causes:**
- LBC IAM role missing permissions (check `kubectl describe ingress` Events for `AccessDenied`)
- ALB still warming up after creation — wait 30–60 seconds for DNS propagation
- ALB target group health check hitting the wrong path — targets will be marked unhealthy and the ALB will return 502. The ALB defaults to `/` which returns 404 on this app. Confirm the healthcheck annotation is present:
  ```bash
  kubectl get ingress user-app -o jsonpath='{.metadata.annotations}'
  # Must include: alb.ingress.kubernetes.io/healthcheck-path: /user-app/health
  ```
- `ingress.groupName` in values does not match other apps in the IngressGroup — the shared ALB may not be routing to this app

### Stale cached image after re-push

**Symptom:** Re-pushed image to ECR under the same tag; pods still run old code.

**Fix:** Always use unique git SHA tags. Never reuse a tag:
```bash
TAG=$(git rev-parse --short HEAD)
docker build --platform linux/arm64 -t ${IMAGE}:${TAG} .
docker push ${IMAGE}:${TAG}
helm upgrade user-app ./charts/user-app \
  --set image.repository=${IMAGE} --set image.tag=${TAG}
```

### Rolling update deadlock (legacy — now prevented by chart config)

> The `maxSurge: 0, maxUnavailable: 1` strategy in `values.yaml` prevents this automatically. Retained for reference and manual recovery if the strategy is ever changed.

**Symptom:** `helm upgrade` hangs. New pod stays `Pending`, old pods not evicted. Karpenter logs `unsatisfiable topology constraint`.

**Cause:** `maxSurge > 0` causes Kubernetes to try creating a new pod before evicting old ones. With strict anti-affinity and 3 AZs fully occupied, there is nowhere to schedule the surge pod.

**Manual recovery:**
```bash
kubectl get replicasets -l app.kubernetes.io/name=user-app
# Note the new ReplicaSet hash, then delete only old pods:
kubectl delete pods -l app.kubernetes.io/name=user-app,pod-template-hash!=<new-hash>
```
